Исправь логику уведомлений и жизненный цикл чатов. Сейчас есть 2 критичные проблемы в операционной логике CRM:

1. После просмотра уведомление не исчезает / не помечается корректно
2. После закрытия диалога в него больше нельзя писать, даже если клиент написал повторно

Нужно исправить это системно.

==================================================
1. УВЕДОМЛЕНИЕ ПОСЛЕ ПРОСМОТРА ДОЛЖНО ИСЧЕЗАТЬ ИЛИ ПОМЕЧАТЬСЯ КАК ПРОЧИТАННОЕ
==================================================

Сейчас уведомления должны вести себя логично.

Правильная логика:
- пришло новое уведомление
- менеджер его открыл / просмотрел
- уведомление исчезает из списка новых
или
- явно помечается как прочитанное
- badge count уменьшается
- счетчики в меню пересчитываются

Нужно реализовать нормальный notification lifecycle.

### Что должно работать
Для уведомлений:
- новый заказ аренды
- новый заказ продажи
- новое бронирование тура
- новый диалог
- новое сообщение
- системные alerts, если используются

### Требования
1. Уведомление должно иметь статус:
   - unread
   - read
   - archived / dismissed, если поддерживается
2. После открытия уведомления:
   - оно должно помечаться как прочитанное
3. Badge counters должны уменьшаться сразу
4. Если логика UI предполагает “новые” отдельно:
   - уведомление должно исчезать из new/unread списка
5. Если есть список “все уведомления”:
   - оно может там оставаться как read, но не как unread

### Нужно проверить
- notification_logs
- notification UI state
- unread counters
- sidebar badge counts
- read/unread marking endpoint
- persistence in DB
- refresh behavior
- re-fetch after read action

### Нужно сделать
- mark as read on open
или
- explicit “Пометить прочитанным”, если нужно
но unread count и визуальный new state должны исчезать после просмотра

==================================================
2. ПОСЛЕ ЗАКРЫТИЯ ДИАЛОГА КЛИЕНТ ДОЛЖЕН МОЧЬ НАПИСАТЬ СНОВА
==================================================

Сейчас после закрытия диалога нельзя больше писать в него, даже если клиент пришел повторно. Это неправильная логика для сайта.

Правильная логика должна быть такой:

Если диалог закрыт:
- менеджер считает обращение завершенным
- но если клиент пишет снова, система должна:
  либо
  1. переоткрыть этот же диалог
  либо
  2. создать новый диалог и связать его с тем же клиентом / external id

Нельзя делать “мертвый” диалог, в который пользователь больше никогда не может написать.

==================================================
3. НУЖНО ОПРЕДЕЛИТЬ ЯВНУЮ ЛОГИКУ СТАТУСОВ ЧАТА
==================================================

Статусы:
- new
- open
- waiting_for_manager
- waiting_for_customer
- closed
- archived

Предлагаемая логика:
- closed = диалог завершен, но может быть reopened при новом сообщении клиента
- archived = скрыт/убран из активной работы, но при новом сообщении также должен либо reopen, либо создавать новый thread по правилам системы

Нужно выбрать и реализовать одну понятную стратегию:

### Вариант A — reopen same session
Если клиент пишет после closed:
- session.status меняется обратно на open / waiting_for_manager
- новое сообщение добавляется в этот же диалог
- менеджер получает уведомление о новом сообщении
- диалог снова появляется в активных

### Вариант B — create new session
Если клиент пишет после closed:
- старая session остается closed
- создается новая session
- она связывается с тем же external id / customer
- менеджер видит новый диалог

Для текущего проекта обычно логичнее:
- closed диалог при новом сообщении клиента автоматически re-open
