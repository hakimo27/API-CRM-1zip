Исправь CRM так, чтобы ею реально можно было пользоваться в работе, а не только смотреть списки. Сейчас не хватает возможности нормально редактировать действующие сущности, в первую очередь заказы. Это критично.

Нужно довести систему до состояния, где менеджер и администратор могут полноценно работать с:
- действующими заказами аренды
- действующими заказами продажи
- бронированиями туров
- товарами
- клиентами
- статусами

Это касается:

- sale orders
- tour bookings

Нужно сделать полноценную карточку заказа с возможностью редактирования.

Для rental orders должно быть можно:
- менять статус заказа
- менять даты аренды
- менять доставку / самовывоз
- менять точку самовывоза
- менять адрес доставки
- менять клиента
- менять комментарий
- менять состав заказа
- добавлять и удалять позиции
- менять количество по позиции
- пересчитывать сумму
- пересчитывать залог
- пересчитывать доступность
- продлевать заказ
- отменять заказ
- завершать заказ
- смотреть историю изменений

Для sale orders должно быть можно:
- менять статус заказа
- менять payment status
- менять delivery method
- менять адрес
- менять клиента
- менять состав заказа
- менять количество
- пересчитывать сумму
- отменять заказ
- завершать заказ
- смотреть историю

Для tour bookings должно быть можно:
- менять статус бронирования
- менять количество участников
- менять дату тура
- менять клиента / контакты
- менять комментарий
- менять сумму / депозит, если это разрешено
- отменять / подтверждать
- смотреть историю

3. РЕДАКТИРОВАНИЕ СОСТАВА ЗАКАЗА
==================================================

В действующем заказе должно быть можно менять состав.

Нужно:
- добавить позицию
- удалить позицию
- изменить количество
- заменить товар
- изменить цену вручную, если есть право
- пересчитать итог
- перепроверить доступность

Для rental:
- при изменении состава заказа availability должна пересчитываться
- reservations должны обновляться
- если появляется конфликт по датам — система должна это показать

Для sale:
- при изменении состава заказа должна пересчитываться total amount
- stock availability должна пересчитываться, если реализована

4. РЕДАКТИРОВАНИЕ ДАТ И ПРОДЛЕНИЕ ЗАКАЗА
==================================================

Для rental orders это критично.

Нужно:
- менять дату начала
- менять дату окончания
- продлевать заказ отдельным действием
- пересчитывать сумму
- пересчитывать резервы
- проверять пересечения с другими бронями
- если конфликт — не давать сохранить или переводить в ручную проверку менеджера

После изменения дат:
- order сохраняется
- reservations обновляются
- availability пересчитывается
- history/log записывается

11. ПРАВА И РОЛИ ПРИ РЕДАКТИРОВАНИИ
==================================================

Учти роли.

Admin:
- может все

Manager:
- может редактировать заказы
- менять статусы
- менять состав заказа
- менять даты
- работать с клиентами
- но не все destructive actions без ограничений

Warehouse:
- может менять статусы сборки / выдачи / возврата
- не должен свободно менять финансы

Нужно:
- permissions checks в UI и backend
- недоступные действия скрывать или дизейблить
- backend все равно валидирует права

13. ЛОГИКА ЖИЗНЕННОГО ЦИКЛА ЗАКАЗА ДОЛЖНА СОХРАНИТЬСЯ
==================================================

Редактирование не должно ломать бизнес-логику.

При изменениях:
- reservations обновляются
- availability пересчитывается
- stock / quantity checks сохраняются
- уведомления отправляются, если нужно
- audit/history пишется
- completed order освобождает арендуемый товар
- cancelled order снимает резерв
- extended order расширяет блокировку по датам
