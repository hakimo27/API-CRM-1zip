Исправь критичный баг в CRM: при ручном создании заказа аренды стоимость считается только после сохранения уже оформленного заказа, а должна считаться прямо в форме создания заказа до сохранения.

Сейчас это ломает работу менеджера:
- он не видит сумму заранее
- не понимает, корректно ли выбраны даты и тариф
- не может нормально согласовать цену с клиентом до создания заказа

Нужно исправить это системно.

==================================================
1. ПРОБЛЕМА
==================================================

Сейчас в ручной форме создания rental order:
- менеджер выбирает товар
- выбирает даты
- выбирает количество
- но стоимость не считается сразу
- цена становится видна только после того, как заказ уже создан

Так быть не должно.

Правильное поведение:
- стоимость должна считаться прямо в create form
- до нажатия “Создать заказ”
- при любом изменении товара / дат / количества
- с учетом тарифа и залога

==================================================
2. ЧТО НУЖНО СДЕЛАТЬ
==================================================

Нужно сделать live pricing в форме создания rental order.

При изменении:
- товара
- даты начала
- даты окончания
- количества
- тарифа, если он выбирается вручную
- доставки/самовывоза, если влияет на цену

система должна автоматически пересчитывать:
- цену по позиции
- выбранный тариф
- залог
- subtotal
- total

И показывать это в UI до сохранения заказа.

==================================================
4. ПРАВИЛЬНАЯ ЛОГИКА
==================================================

Форма создания rental order должна работать так:

Шаг 1:
- выбрать клиента

Шаг 2:
- выбрать даты аренды

Шаг 3:
- выбрать товар

Шаг 4:
- выбрать количество

После этого автоматически:
- определяется подходящий тариф
- показывается название тарифа
- считается цена по позиции
- считается залог
- считается total заказа

Если есть несколько позиций:
- сумма считается по всем позициям
- общий итог пересчитывается автоматически

Если тариф не найден:
- не показывать просто 0
- показать понятное предупреждение:
  - “Для выбранного товара не найден подходящий тариф”
  - “Невозможно рассчитать цену: проверьте даты или тарифы”

==================================================
5. 0 НЕ ДОЛЖЕН БЫТЬ ТИХИМ FALLBACK
==================================================

Сейчас, если расчет не произошел, нельзя просто оставлять 0 как будто это валидная цена.

Нужно:
- убрать silent fallback в 0
- если цена реально не рассчитана, показывать explicit state:
  - “Цена не рассчитана”
  - “Нет подходящего тарифа”
  - “Заполните даты аренды”
- 0 допустим только если это реально бесплатная услуга, а не ошибка расчета
